> **Attempts:**
> - Scope creep: when demoing the `contentLeft → contentRight` swap with the vendor Icon, I introduced a function-form `props: (base) => ...` in the HOC and used it in Hero to avoid duplicating the Icon element. The user never asked for it — they wanted the Icon rendered twice, trivial JSX duplication.
> - Diagnosis 1 (commit `8096b95`): claimed the type error came from the union-typed `props`, flattened `BreakpointOverride`. Did not fix the error. Commit message was wrong.
> - Diagnosis 2: proposed generic inference on `withBreakpoints(BaseTile)` as the cause, but without running diagnostics. Another guess.
>
> **Actual cause:** Confirmed via `tsc --noEmit` in nice-website-2025 (TypeScript `^4.9.5`, `strict: true`): TS 4.9 fails to infer `P` through the `React.ComponentType<P>` argument constraint when `withBreakpoints(BaseTile)` is called without an explicit type argument. `Tile` collapses to a weak type, `breakpoints` loses its `P`, and the arrow-function parameter has no contextual type → `TS7006` under `noImplicitAny`. Fix: annotate the call site as `withBreakpoints<LocalTileProps>(BaseTile)`.
>
> **Rule for next time:** Stop at one failed fix, not two. When a type error persists after a proposed fix, run diagnostics (`tsc --noEmit`, `tsc --traceResolution`, probe types with `type X = ...` snippets) before proposing another cause. Do not add features the user did not request — especially features that "save a few characters" at the cost of new API surface.
